🔴 T-001: Санитизация пользовательского ввода (XSS)

Категория: Безопасность
Сложность: 2/5
Время: ~2 часа

Описание:
Все пользовательские поля (name, class_name, error_description, fact, error_text, correct_text) могут содержать вредоносный HTML/JavaScript.

Текущая проблема:
javascript

// Прямая вставка без санитизации
document.getElementById('profileFactBox').innerHTML = fact;

Решение:
Внедрить функцию sanitizeHTML() и применять её во всех местах рендеринга.

Критерии приёмки:

    Создана функция sanitizeHTML(str)

    Заменены все innerHTML на sanitizeHTML()

    Протестирован ввод <script>alert('XSS')</script>

    Спецсимволы отображаются как текст

Пример кода:
javascript

function sanitizeHTML(str) {
    if (!str) return '';
    const map = {
        '&': '&amp;', '<': '&lt;', '>': '&gt;',
        '"': '&quot;', "'": '&#x27;', '/': '&#x2F;'
    };
    return str.replace(/[&<>"'/]/g, s => map[s]);
}

🔴 T-002: Защита от SQL-инъекций

Категория: Безопасность
Сложность: 3/5
Время: ~3 часа

Описание:
Некоторые запросы используют конкатенацию строк, что делает систему уязвимой для SQL-инъекций.

Проблемные места:
javascript

// ❌ Опасно
db.exec(`SELECT * FROM students WHERE name = '${name}'`);
db.exec(`DELETE FROM students WHERE id = ${id}`);

Решение:
Перевести все запросы на параметризованные (? или :param).

Критерии приёмки:

    Найдены все уязвимые запросы

    Заменены на db.run('... WHERE name = ?', [name])

    Везде используется prepare() + getAsObject() где возможно

    Проверка на ' OR '1'='1 не даёт результатов

🔴 T-003: Транзакционная обёртка для импорта

Категория: Безопасность
Сложность: 3/5
Время: ~2.5 часа

Описание:
При импорте JSON, если один из элементов вызывает ошибку, БД остаётся в частично изменённом состоянии.

Решение:
Создать функцию safeTransaction(callback) с BEGIN/COMMIT/ROLLBACK.

Критерии приёмки:

    Функция safeTransaction() создана

    Импорт обёрнут в транзакцию

    При ошибке все изменения откатываются

    Пользователь видит уведомление об ошибке

🔴 T-004: Защита от дублирования записей

Категория: Безопасность
Сложность: 2/5
Время: ~1.5 часа

Описание:
Сейчас UNIQUE ограничение есть, но ошибка не обрабатывается gracefully. Пользователь видит сбой без понятного сообщения.

Решение:
Перехватывать исключение UNIQUE constraint failed и показывать понятное уведомление с предложением редактировать.

Критерии приёмки:

    Перехват ошибки в submitStudentForm()

    Показ уведомления: «Ученик уже существует»

    Автоматический переход в режим редактирования

    Тест: попытка создать дубликат
