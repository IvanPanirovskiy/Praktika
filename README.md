# Правила проекта 
| Пункт | Что зафиксировано |
| :--- | :--- |
| **1. Отрасль** | Образование (преподавание и учительская деятельность). |
| **2. Тренд** | «Отложенная память» — ИИ как внешний гиппокамп. Учитель теряет контекст каждого ученика из-за перегрузки. |
| **3. Выбранная гипотеза** | «Архитектор ошибочных траекторий» (Анти-RAG для ошибок) — система, которая вычленяет типы ошибок ученика, строит их траекторию во времени и дает учителю готовый план действий (3 фамилии в неделю). |
| **4. Носитель проблемы (кто именно)** | Учитель-предметник (математика, физика, химия, русский язык), 5–8 классы, нагрузка 25+ часов в неделю, проверяет 50–60 учеников, стаж 3–8 лет. |
| **5. Острота проблемы** | 9.5 из 10 (топ-3 причины выгорания и увольнения молодых учителей). |
| **6. Частота** | Ежедневно, 5–6 раз в неделю (каждая проверка тетрадей — момент потери контекста). |
| **7. Тип ошибок, с которыми работаем** | Процедурная, концептуальная, перцептивная, трансферная. |
| **8. Ключевое ограничение (риск)** | Распознавание рукописного ввода (математические символы). На MVP — уйти в цифровой ввод (набор формул). |
| **9. Монетизация (первая гипотеза)** | B2G (лицензия на школу 500k–1M руб/год) или B2B (частные школы 150k–300k руб/год). B2C — только для репетиторов (нужно уточнить). |
| **10. Следующий шаг** | Глубинные интервью с 3 учителями математики в школах-партнерах для валидации гипотезы. |
| **11. Целевая аудитория (кто платит)** | **B2G:** Департаменты образования регионов (госзакупки по программе «Цифровая образовательная среда»). **B2B:** Частные школы и онлайн-школы (Фоксфорд, Сириус, сети лицеев). **B2C:** Репетиторы с нагрузкой 20+ учеников (нужно уточнить готовность платить). Учитель — пользователь, а не плательщик. |
| **12. Персона №1 — Преподаватель** | Алексей, 34 года, учитель математики в московской гимназии, стаж 6 лет, нагрузка 26 часов, 78 учеников. Проверяет тетради 4 раза в неделю (10 часов). Перестал помнить индивидуальные ошибки. Excel бросил через 3 недели. На родительских собраниях дает общие ответы, чувствует вину. Нужны факты для родителей, а не рекомендации. |
| **13. Персона №2 — Ученик** | Полина, 12 лет, 7 класс. Математика — стабильная «3». При застревании — белый шум и избегание. Не перепроверяет себя. Запоминает яркие фразы, а не алгоритмы. На «будь внимательнее» — обида, действий не меняет. Нужна визуализация ошибки и конкретный шаг, где ошиблась. |
| **14. Ценностное предположение (из интервью)** | Учитель купит продукт, если он: (а) не добавляет времени к проверке; (б) дает конкретные факты для родительских собраний; (в) автоматически работает, не требуя ручного ввода. Ученик примет продукт, если он: (а) показывает, где именно ошибка; (б) дает визуальный якорь («понимаю по картинке»); (в) не требует «стать внимательнее» — дает конкретное действие. |
| **15. Роли в команде (гипотеза)** | **Продакт / исследователь** — формулирует гипотезы, ведет интервью, собирает обратную связь. **ML-инженер** — строит пайплайн распознавания/классификации ошибок (сначала на цифровых данных). **Fullstack-разработчик** — делает интерфейс для учителя (дашборд) и интеграцию с ЭЖ/API. **Специалист по педагогике / методист** — разрабатывает типологию ошибок и проверяет, что рекомендации педагогически корректны. (Нужно уточнить, берем ли методиста в штат или внешнего эксперта). |
| **16. Product Vision (утверждено)** | Мы решаем проблему, о которой учителя редко говорят вслух: они перестают видеть прогресс учеников, потому что их рабочая память перегружена. Наш продукт берет на себя функцию внешней памяти — фиксирует каждую ошибку, классифицирует ее тип, показывает динамику и подсвечивает те паттерны, которые учитель не может заметить в потоке 80 проверенных тетрадей за вечер. В итоге учитель снова видит не «гору тетрадей», а конкретных детей с конкретными трудностями. А ученик перестает получать бесполезные «будь внимательнее» — он получает указание на конкретный шаг, где споткнулся. |
| **17. MVP Scope — что делаем** | (1) Импорт цифровых решений (CSV/JSON/XLSX) из существующих платформ через ручной экспорт. (2) Структурирование решения по шагам (фиксация шага с ошибкой). (3) Накопление истории ошибок по ученику. (4) Еженедельный дашборд с тремя блоками: «Красный флаг» (до 5 фамилий), «Факты для родителей» (текст с примерами), «Прогресс» (исчезнувшие ошибки). (5) Экспорт фактов в один клик (готовый текст для копирования). |
| **18. MVP Scope — что НЕ делаем** | Распознавание рукописного текста. Классификацию типа ошибки (процедурная/концептуальная/перцептивная). Рекомендации «что делать с учеником». Автоматическую проверку работ. Прогнозы. Мобильное приложение. Интеграцию с электронными журналами. Мультипредметность. Личный кабинет ученика. Рекомендательную систему для подбора заданий. Систему уведомлений. |
| **19. Ключевой сценарий (Core Interaction)** | Учитель открывает дашборд в понедельник утром. Видит 3 фамилии в блоке «Красный флаг». Кликает на фамилию → видит 2–3 примера ошибок из работ за месяц и готовый текст для родителя. Нажимает «Скопировать текст для родителей» → вставляет в родительский чат → отправляет. Повторяет для остальных фамилий. |
| **20. Стоп-условие гипотезы (MVP fail)** | Если за 2 недели пилота ни один учитель не скопировал текст для родителей хотя бы 1 раз — гипотеза не подтверждена. Мы строим не ту ценность. |

# Журнал решений 
### (запись №1)

**Решение:** Мы выбрали отрасль «Преподавание» и гипотезу «Архитектор ошибочных траекторий». 

**Обоснование:** Среди 6 гипотез эта бьет в самую высокую и частую боль учителя (проверка работ + потеря контекста ошибок ученика), имеет максимальную остроту (9.5 из 10) и при этом не требует сложной интеграции с внешними системами на старте (можно начать с ручного ввода). Также именно эта гипотеза дает долгосрочное конкурентное преимущество — накопление базы когнитивных паттернов, а не просто статистики. 

**Что требует уточнения:** Готовы ли учителя-предметники тратить 2 минуты на ручную разметку типа ошибки в MVP? Будет проверено на глубинных интервью.

## (запись №2)

**Решение:** Интервью и скептический разбор показали, что исходная гипотеза требует кардинальной корректировки по трем направлениям: (1) способ сбора данных — отказ от ручного сканирования, только цифровой ввод; (2) позиционирование — сместить с «типологии ошибок» на «факты для родительских собраний»; (3) каналы продаж — проработать B2B/B2G-стратегию, без этого входа нет.

**Что дали интервью:** Подтвердили остроту боли (учитель не помнит траекторию, Excel провалился, родители хотят конкретики), но добавили жесткие ограничения: дополнительный шаг убивает продукт, учитель не доверяет рекомендациям ИИ, а ученику нужна не типология, а конкретное место ошибки и визуализация.

**Что вскрыл скептик:** Технология распознавания рукописного текста + классификация логики ошибок — нереалистична на старте. Конкуренты (Яндекс, МЭШ) догонят за квартал, если увидят тренд. Отличимость слабая — «типология» продается, только если упакована в понятный школе результат («снижение жалоб», «экономия времени»).

**Скорректировали ли гипотезу и почему:** Да. Гипотеза №1 остается в силе по *проблеме* (диагностика ошибок + траектория), но меняется по *решению*: отказываемся от ручного сканирования на MVP, работаем только с цифровыми решениями (интеграция через API или загрузка файлов), смещаем ценностное предложение с «план действий для учителя» на «факты для родительского собрания».

## (запись №3)

**Решение:** Утвержден MVP Scope с четким разделением «делаем / не делаем». Выделено одно ключевое взаимодействие — копирование готового текста для родителей. MVP строится вокруг одного экрана (дашборд → карточка ученика с текстом → кнопка «Скопировать»).

**Обоснование:** Интервью с Алексеем показали, что его главная боль — не отсутствие аналитики, а чувство вины перед родителями из-за общих ответов. MVP проверяет гипотезу: «Учитель будет пользоваться системой, если она дает готовые факты для родителей без дополнительных действий». Технология распознавания и типология ошибок — вынесены в раздел «НЕ делаем» как отвлекающие факторы. Успех MVP измеряется одним действием — копированием текста. Если учитель копирует — продукт нужен. Если нет — строим не то.

**Что дальше:** Построить черновой макет трех экранов (дашборд → карточка ученика → дашборд после отправки) на основе User Flow из блока 3. Проверить на Алексее в пилотном режиме с имитированными данными (загружены вручную).

## (запись №5)

**Решение:** После подтверждения ценности MVP фокус смещается с проверки гипотезы на устранение главного трения — ручного экспорта данных. Приоритет №1: автоматизация сбора данных через браузерное расширение. Приоритет №2: доработка UX копирования (редактирование тональности). B2B-упаковка откладывается до момента, когда продукт будет работать полностью автономно без ручных действий учителя.

**Обоснование:** Пилот показал, что учитель готов пользоваться продуктом, но тратит 4 минуты в неделю на подготовку данных. Это терпимо для одного учителя, но неприемлемо для масштабирования: если мы продадим решение школе с 50 учителями, ручная работа съест всю экономию времени. Автоматизация сбора данных — это не улучшение, а обязательное условие перехода от пилота к продукту. B2B-продажи без автоматизации невозможны: директор школы не купит инструмент, который требует от учителей дополнительных действий.

**Что требует уточнения:** Какие ЭЖ (электронные журналы) поддерживать в первую очередь? Нужно понять, какие платформы используют целевые школы-партнеры, чтобы не делать расширение "в никуда".

# Ванпейджеры
## Ванпейджер блока 1

Часть А. Что мы сделали и к чему пришли

Исходная точка: Анализ 10 макротрендов — выбрали тренд «Отложенная память» (ИИ как внешний гиппокамп). Суть: человек перестает удерживать контекст и причинно-следственные связи, потому что ИИ дает ответ быстрее, чем мозг успевает включить критическое мышление.

Сужение до отрасли: Преподавание и учительская деятельность.

    Учитель — классический носитель «перегруженного гиппокампа»: 30+ учеников, 100+ работ в неделю, физически не может помнить траекторию ошибок каждого.

    В образовании это бьет не в скорость, а в качество: учитель перестает видеть структуру мышления ученика, дает шаблонную обратную связь → ошибки закрепляются на годы.

Генерация гипотез: Сформулировали 6 продуктовых гипотез, каждая с привязкой к конкретному носителю боли.

Выбор: Остановились на Гипотезе №1 — «Архитектор ошибочных траекторий» (Анти-RAG для ошибок).

Часть Б. Как это задает весь проект

Этот выбор определяет генеральную линию всего проекта:

    Мы НЕ делаем очередную LMS, конструктор уроков или чат-бота для родителей.

    Мы делаем диагностическую ИИ-систему, которая работает на уровне типологии когнитивных ошибок (процедурная / концептуальная / перцептивная / трансферная).

    Главный пользователь — не ученик, не родитель, а учитель-предметник с нагрузкой 25+ часов (сегмент 3–8 лет стажа).

    Главная метрика — не скорость проверки, а снижение повторяемости ошибок у конкретного ученика и возвращение учителю радости видеть прогресс.

    Долгосрочная ставка — накопление базы паттернов ошибок, которая позволит в будущем автоматически подбирать упражнения под конкретный когнитивный сбой (выход в адаптивное обучение).

Ключевой принцип, заданный этим блоком:

    «Мы не автоматизируем рутину ради автоматизации. Мы восстанавливаем связь между учителем и учеником, которую убила перегрузка памяти».

Отличимость слабая — «типология ошибок» это фича, а не рыночное позиционирование. |

## Ванпейджер блока 3

**Часть А. Что мы сделали и к чему пришли**

**Диагностика:** На основе утвержденного Product Vision и данных интервью (блок 2) мы жестко ограничили MVP Scope. Определили, что продукт делает, что осознанно не делает, и какой критерий добавления новых функций.

**Ключевое решение:** Выделили одно ключевое взаимодействие, ради которого существует продукт — учитель открывает дашборд в понедельник утром и за 3 минуты получает готовые факты для родительского собрания. Всё остальное — подготовка к этому моменту.

**Что сделали:**
- Сформулировали MVP Scope с тремя разделами (делаем / не делаем / критерий добавления фичи).
- Описали ключевой пользовательский сценарий по шагам: от открытия дашборда до копирования текста для родителя.
- Построили User Flow с тремя экранами (дашборд → детальная карточка ученика → дашборд после отправки).
- Определили единственное, что строим завтра — интерфейс дашборда с готовыми текстами для родителей.

**К чему пришли:** MVP — это не система распознавания ошибок и не аналитическая платформа. Это **генератор готовых текстов для родителей**, который автоматически собирает факты об ошибках ученика. Если учитель копирует текст и отправляет родителям — продукт работает. Если нет — мы строим не то.

**Часть Б. Как это меняет весь проект**

| Что было | Что стало | Как меняет |
| :--- | :--- | :--- |
| MVP должен был анализировать ошибки и давать план действий | MVP дает готовые тексты для родителей, а не план действий | Меняем ценностное предложение: не «помогаем учителю учить», а «помогаем учителю говорить с родителями». ### Бэклог следующих шагов (Sprint 2)

На основе результатов пилота и инсайтов формируем бэклог. Приоритет отдается устранению ручного труда и подготовке к продажам.

**1. Автоматизация сбора данных (Снятие трения Data Friction)**
*   *Задача:* Избавить учителя от ручного экспорта CSV.
*   *Решение:* Разработать легкое браузерное расширение (или скрипт-парсер), которое само забирает данные из открытых API/веб-интерфейсов популярных ЭЖ (Дневник.ru, Skysmart, Учи.ру) и пушит их в наш дашборд по расписанию (например, в воскресенье вечером).
*   *Критерий успеха:* Учитель тратит 0 минут на подготовку данных.

**2. Доработка UX «Копирования» (Реакция на инсайт пилота)**
*   *Задача:* Дать учителю контроль над тоном коммуникации.
*   *Решение:* Заменить жесткую кнопку «Скопировать» на модалку с предзаполненным текстом, где есть 3 переключателя тональности: «Строгий/Официальный», «Мягкий/Поддерживающий», «Только факты». Добавить возможность быстрого inline-редактирования перед отправкой в буфер обмена.

**3. Подготовка B2B/B2G «Упаковки» (Переход к плательщику)**
*   *Задача:* Перевести ценность с языка учителя на язык директора школы / завуча.
*   *Решение:* Сформировать метрики для покупателя. Мы не можем продать «дашборд Алексея». Мы продаем: «Снижение количества жалоб от родителей на некомпетентность учителей на X%», «Экономия Y часов педсостава на родительских чатах», «Инструмент для завуча по точечной работе с отстающими (Красный флаг)».
*   *Артефакт:* Подготовить одностраничный Case Study по пилоту с Алексеем для показа первым B2B-клиентам (частные школы).

**4. Расширение пилота (Валидация на других предметах)**
*   *Задача:* Проверить, работает ли логика «Красного флага» вне математики.
*   *Решение:* Подключить к пилоту еще 2 учителей (физика, русский язык). 
*   *Риск:* В математике ошибка — это конкретный шаг. В русском языке или литературе «ошибка» может быть более размытой. Нужно проверить, как наш алгоритм структурирует «цифровой след» по гуманитарным предметам, чтобы генерировать адекватные факты для родителей.

**5. Возврат к Персоне №2 (Ученик) — Отложенная задача**
*   *Задача:* Дать Полине «визуальный якорь» до наступления белого шума.
*   *Решение (на будущее):* После того как мы закроем боль учителя и настроим авто-сбор данных, мы сможем генерировать для ученика микро-карточки «Твоя суперсила на этой неделе: перенос скобок». Но это *строго после* того, как продукт станет прибыльным за счет B2B/B2G. Сейчас фокус только на учителе.
| Предполагали, что учитель будет фотографировать/сканировать работы | В MVP — только цифровой ввод через импорт файлов | Снимаем технологический риск (распознавание рукописного текста) на старте. Проверяем гипотезу о ценности траектории, а не о технологии. |
| Строили систему для учителя | Строим систему, которая дает учителю готовый текст, который он копирует и вставляет в родительский чат | Меняем интерфейс: вместо аналитических дашбордов и графиков — одна страница с текстом и одной большой кнопкой «Скопировать». |
| Рассматривали B2G и B2B каналы | MVP проверяем на конкретном учителе (Алексей) без продаж | Сужаем фокус: сначала проверить, пользуется ли хотя бы один учитель. Если нет — остальные каналы не важны. |
